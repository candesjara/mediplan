## FASE 1 - Identificar y Clasificar Endpoints

## 1) Listado de endpoints actuales

### Modulo Auth
- `POST /auth/register`
- `POST /auth/login`

### Modulo Doctores
- `GET /doctores`
- `POST /doctores`
- `GET /doctores/protegida`

### Modulo Pacientes
- `GET /pacientes`
- `POST /pacientes`
- `PUT /pacientes/:id`
- `DELETE /pacientes/:id`

### Modulo Citas
- `GET /citas`
- `POST /citas`
- `PUT /citas/:id`
- `DELETE /citas/:id`

## 2) Clasificacion

### A) CRUD (administracion de informacion)
- Doctores: `GET /doctores`, `POST /doctores`
- Pacientes: `GET /pacientes`, `POST /pacientes`, `PUT /pacientes/:id`, `DELETE /pacientes/:id`
- Citas: `GET /citas`, `POST /citas`, `PUT /citas/:id`, `DELETE /citas/:id`
- Auth (gestion de usuarios/sesion): `POST /auth/register`, `POST /auth/login`

### B) Acciones del negocio
- Agendar cita: `POST /citas` (crea cita futura valida)
- Confirmar cita: `PUT /citas/:id` con `estado: "confirmada"`
- Cancelar cita: `PUT /citas/:id` con `estado: "cancelada"`
- Validar acceso autenticado: `GET /doctores/protegida`

## 1. Registrar Usuario

Nombre: Registrar usuario  
URL: `/api/auth/register`  
Metodo: `POST`  
Que hace: Registra un usuario nuevo con password hasheada.  
Protegido: No  
Rol requerido: No aplica  
Body esperado:

```json
{
  "nombre": "Admin MediPlan",
  "correo": "admin@mediplan.com",
  "password": "Admin123",
  "rol": "admin"
}
```

Respuesta exitosa:

```json
{
  "message": "Usuario registrado exitosamente.",
  "usuario": {
    "id": 1,
    "nombre": "Admin MediPlan",
    "correo": "admin@mediplan.com",
    "rol": "admin"
  }
}
```

Posibles errores:

```json
{
  "message": "El correo ya esta registrado."
}
```

```json
{
  "message": "Error en el registro",
  "error": {}
}
```

Codigo HTTP esperado: `201`, `400`, `500`

## FASE 4 - Validacion Contra Checklist Oficial

Checklist:

- ✔ Login funcional
- ✔ Roles existentes (`admin`, `doctor`, `paciente` en modelo y token)
- ✔ CRUD de entidades (Doctores, Pacientes, Citas)
- ✔ Acciones del negocio (agendar, confirmar, cancelar)
- ✔ Cambio de estados en citas
- ✔ Consulta especial (`GET /api/doctores/protegida`)
- ✔ Pruebas documentadas (unitarias e integracion + guia Postman)

Hallazgos y faltantes menores:
- El rol existe pero **no se aplica autorizacion por rol** en rutas protegidas (solo token).
- `GET /api/doctores` y `POST /api/doctores` son publicos; para entorno real deberian protegerse.
- No hay endpoint de consulta filtrada por rango de fechas (seria una mejora funcional simple).

Mejoras simples sugeridas (sin cambiar arquitectura):
1. Agregar middleware de rol simple (`soloAdmin`) en rutas criticas.
2. Proteger `POST /api/doctores`.
3. Crear una consulta especial sencilla: `GET /api/citas?estado=pendiente`.
